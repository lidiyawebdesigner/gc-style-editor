# GC Pipeline — команда расширения GetCourse Style Editor

Мультиагентный конвейер под реальную работу Лиды над расширением. Раньше она гоняла одного агента по кругу (придумай → напиши CSS → нет, не так → проверь → всё равно не применилось на проде). Здесь работа разнесена по ролям с гейтами качества: дальше по конвейеру результат не идёт, пока не прошёл ревью и тест.

## Команда

| Агент | Зона |
|---|---|
| `gc-feature-ideator` | идеи по фичам приложения |
| `gc-ux-designer` | визуальный стиль панели + UX/UI расширения |
| `gc-css-dev` ⭐ | премиальный рабочий CSS для блоков GC + логика экспорта |
| `gc-qa-reviewer` | ревью перед сдачей (гейт), может вернуть на доработку |
| `gc-tester` | тест на тестовых блоках `brief/` + 4 брейкпоинта (гейт) |
| `qa-processor` | разбор заметок Лиды из расширения в список задач |

## Ответы на вопросы урока

- **Что на выходе:** готовый артефакт по типу задачи — рабочий CSS для блока GC (с указанием куда вставлять), либо правки кода/стиля расширения, прошедшие ревью и тест, с финальным отчётом «готово».
- **Какие этапы:** идея/дизайн (опционально) → разработка → ревью (гейт) → тест (гейт) → сдача. Гейты с возвратом: ревью или тест могут отбросить результат назад к разработчику.
- **Какие роли:** идейщик фич, UX/UI-дизайнер панели, спец по GC-CSS, QA-ревьюер, тестировщик, плюс приёмщик фидбека (qa-processor). У каждого свой `<quality_checklist>`.
- **Как передаются результаты:** напрямую между этапами — разработчик отдаёт CSS/код + «куда вставлять», ревьюер выносит PASS/FAIL с конкретикой, тестер — матрицу по брейкпоинтам. Оркестратор держит петли и не теряет контекст между ними.
- **Как проверяется качество:** два независимых гейта (ревью кода/CSS + тест на реальной разметке и экранах) до того, как Лиде скажут «готово». FAIL на любом — возврат к разработчику, не к Лиде.

## Тип пайплайна: смешанный

- **Параллельно** (по необходимости): `gc-feature-ideator` и `gc-ux-designer` на старте, если задача про новое направление.
- **Последовательно с гейтами и петлями:** разработка → ревью → тест → сдача.

```
            ┌─ gc-feature-ideator ─┐   (опц., параллельно)
   задача ──┤                      ├─▶ gc-css-dev / gc-ux-designer
            └─ gc-ux-designer ─────┘            │
                                                ▼
                                       gc-qa-reviewer ──FAIL──▶ назад к разработке
                                                │PASS
                                                ▼
                                          gc-tester ──FAIL──▶ назад к разработке
                                                │PASS
                                                ▼
                                       «готово» + отчёт Лиде
```

## Маршрутизация (как оркестратор выбирает агентов)

1. **Разбери тип задачи:**
   - идея/«куда развивать» → `gc-feature-ideator`
   - вид/UX панели расширения → `gc-ux-designer` → dev → ревью → тест
   - CSS для блока GC / «не применяется» / пресет → `gc-css-dev` → ревью → тест
   - разбор заметок из расширения → `qa-processor` (отдельно, выдаёт задачи, которые потом идут в этот конвейер)
2. **Разработка:** запусти нужного разработчика (`gc-css-dev` и/или `gc-ux-designer`) через Agent tool с `subagent_type`.
3. **Гейт ревью:** запусти `gc-qa-reviewer`. Если FAIL — верни список правок разработчику и повтори. Не пропускай дальше с FAIL.
4. **Гейт тест:** запусти `gc-tester`. Если FAIL — назад к разработке. Если нужен реальный браузерный клик-тест — сообщи Лиде про Playwright.
5. **Сдача:** только после PASS на обоих гейтах отчитайся Лиде: что сделано, артефакт/CSS, куда вставлять, что проверить руками, остаточные риски.

## Заметки

- Запускать субагентов через Agent tool с `subagent_type` (`gc-css-dev`, `gc-qa-reviewer` и т.д.), а не делать их работу самому — смысл конвейера в делегировании и независимых гейтах.
- Лимит петель: если после 2 возвратов гейт всё ещё FAIL — стоп, вынеси проблему Лиде, не крути бесконечно.
- Все правила GC-CSS — внутри системника `gc-css-dev` (`<gc_css_rules>`) и в `CLAUDE.md` проекта.
